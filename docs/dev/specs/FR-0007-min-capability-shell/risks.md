# FR-0007 风险

FR-0007 定义的是 Phase 1 到 Phase 2 之间的共享能力壳边界，属于公共契约基座风险项，需要单独收口。

当前 formal 承接 issue 为 `#360`；历史 `#159` 仅保留来源引用，不再作为 closeout 事实来源。

## 关键风险

| 风险 | 触发条件 | 影响 | 缓解手段 | 观察信号 | 回滚 / 收口方式 |
|---|---|---|---|---|---|
| 能力壳与 FR-0001 外层壳冲突 | 在能力层引入第二套入口或第二套返回壳 | 上层调用方解析逻辑分叉，契约不可复用 | 明确“外层壳不变，能力壳内嵌”原则；契约测试覆盖 | 同一命令出现两种返回结构 | 阻断合并，回到单壳结构 |
| 能力壳与 FR-0004 诊断壳混写 | 把 `error.diagnosis`、`observability` 与 `error.details` / `capability_result` 混成同一层 | 诊断语义与能力语义互相覆盖，调用方无法区分“为什么失败”和“哪个能力失败” | 明确 FR-0004/FR-0007 各自字段归属；补充共存测试 | 错误响应中出现重复阶段字段或观测字段被塞入 `summary.capability_result` | 回退到“诊断壳在 FR-0004、能力壳在 FR-0007”的分层结构 |
| 能力响应被 FR-0006 证据层反向塑形 | 为了落库方便，在响应层引入持久化专属字段或要求“先落库再返回” | 响应契约耦合持久化实现，SQLite 故障会放大成能力层协议漂移 | 明确“FR-0006 只消费映射，不反向定义响应”原则；评审时检查是否新增持久化字段 | 响应开始依赖 `record_id`、`event_id` 等持久化字段，或落库失败直接改变能力壳结构 | 阻断变更，恢复响应层与证据层解耦 |
| 输入字段漂移 | 不同能力随意改写 `ability/input/options` 语义 | 首个平台样本仍是一次性特例，后续无法统一复用 | 冻结最小字段字典与校验规则 | 同一能力同版本字段名变化 | 按契约回收字段并补兼容说明 |
| 失败语义被压平 | 所有失败都归并为单一错误文本 | 无法判断重试、修复或人工介入路径 | 统一 `error.details.stage/reason` | 错误只剩 message 文本，无结构字段 | 回退到统一错误映射层 |
| 输出壳缺失或混入大体量数据 | 直接透传原始结果，不组装 `capability_result` | 上层上下文成本失控，信息披露失效 | 强制 `summary.capability_result` 校验；坚持摘要返回 | `stdout` 返回体积异常或缺少 `capability_result` | 阻断发布并恢复摘要壳 |
| Scope Creep 过早扩张能力系统 | 在 FR-0007 引入导入/分发/版本仓库 | Phase 1 关键闭环延迟，评审焦点失真 | 严格执行非目标；将扩张项移入后续 FR | 规约出现市场化或平台化能力管理条目 | 从 FR-0007 剥离并独立立项 |

## Stop-Ship 条件

出现以下任一情况时，不应进入实现或合并：

- 能力调用未复用 FR-0001 外层壳
- FR-0004 / FR-0006 的字段边界被混入 FR-0007 响应契约
- `summary.capability_result` 最小必填字段未冻结
- `error.details` 缺失 `ability_id/stage/reason` 最小结构
- 能力壳提案混入能力分享、导入、版本分发等超范围内容

## 剩余风险

- 后续 FR 可能需要更细粒度的能力版本策略；当前阶段只冻结最小壳，不处理迁移工具链。
- 不同平台的能力私有字段差异仍可能带来认知成本，需在 Phase 2 再统一描述层。
- issue `#360` 当前仍保留历史 `Refs #154` 文本，后续若 issue 描述更新，应保持与本套件“FR-0004 以后继 `#357` 为正式边界”的表述一致。
