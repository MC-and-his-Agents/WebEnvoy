# FR-0018 研究记录

## 研究问题

在 Phase 2 中，如何让“已保存能力”具备最小验证、最小重放和最小可信判断，而不把它提前扩张成完整版本治理、导入/安装或自动修复系统。

## 当前输入

- `vision.md` 中“帮助用户生产、验证、维护能力的平台”定位
- `docs/dev/roadmap.md` 中对最小验证、重放与可信判断的要求
- `#155` issue 范围
- `FR-0017` 对候选能力描述的正式冻结

## 收敛结论

1. 候选能力描述与验证结果必须分层，否则“已保存”和“已验证”会被混淆。
2. 健康状态需要是用户可读的大状态，而不是低层错误码镜像。
3. 重放是最小再运行能力，不是自动修复。
4. 最近一次验证结果必须继续引用既有运行证据，而不是新建第二套真相源。
