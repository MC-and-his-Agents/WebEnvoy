# FR-0019 风险与回滚

## 主要风险

### 1. 一次成功被误写成正式可复用

- 风险：L2 在未知网站上暂时成功一次，就被误当成已验证或正式能力。
- 缓解：
  - 强制 `candidate_shell_seed`
  - 强制区分首次成功与后续验证

### 2. L2 失败被伪装成“部分成功”

- 风险：缺少结构化成功判定时，失败路径也可能被包装成可用结果。
- 缓解：
  - 冻结最小成功条件
  - 冻结失败大类

### 3. 越界到完整 L1 或完整 L2 平台

- 风险：规约阶段过早把 L2 首次可用扩张成完整通用平台，或把 L1 兜底细节混进来。
- 缓解：
  - 在 `spec.md`、`plan.md` 中明确非目标与实现约束

### 4. handoff 输出不足以进入候选能力链路

- 风险：L2 首次可用虽然有结果，但无法沉淀成候选能力。
- 缓解：
  - 冻结 `candidate_shell_seed`
  - 明确其与 `FR-0017` 的映射边界

### 5. XHS 专用 gate 被误继承到未知网站

- 风险：把 `FR-0010/0011` 的 XHS 专用门禁条件直接当成未知网站 L2 的正式前置，导致范围与平台边界混淆。
- 缓解：
  - 只继承站点无关的最小风险门禁原则
  - 在本 FR 中显式排除 XHS 专用 gate 条件与账号特有审批路径

## 回滚原则

- 若 reviewer 认为 L2 首次可用当前仍证据不足，应优先回退范围，保留“最小读取 + handoff 输出”，不要把更多临时实验写成正式契约。
